1 /++
2 This module is implementing the Expected idiom.
3 
4 See the [Andrei Alexandrescu’s talk (Systematic Error Handling in C++](http://channel9.msdn.com/Shows/Going+Deep/C-and-Beyond-2012-Andrei-Alexandrescu-Systematic-Error-Handling-in-C)
5 and [its slides](https://skydrive.live.com/?cid=f1b8ff18a2aec5c5&id=F1B8FF18A2AEC5C5!1158).
6 
7 Or more recent ["Expect the Expected"](https://www.youtube.com/watch?v=nVzgkepAg5Y) by Andrei Alexandrescu for further background.
8 
9 It is also inspired by C++'s proposed [std::expected](https://wg21.link/p0323) and [Rust's](https://www.rust-lang.org/) [Result](https://doc.rust-lang.org/std/result/).
10 
11 Similar work is [expectations](http://code.dlang.org/packages/expectations) by Paul Backus.
12 
13 Main differences with that are:
14 
15 $(LIST
16 * lightweight, no other external dependencies
17 * allows to use same types for `T` and `E`
18 * allows to define $(LREF Expected) without value (`void` for `T`)
19 * provides facility to change the $(LREF Expected) behavior by custom `Hook` implementation using the Design by introspection.
20 )
21 
22 Default type for error is `string`, i.e. `Expected!int` is the same as `Expected!(int, string)`
23 
24 $(LREF Expected) has customizable behavior with the help of a third type parameter,
25 `Hook`. Depending on what methods `Hook` defines, core operations on the
26 $(LREF Expected) may be verified or completely redefined.
27 If `Hook` defines no method at all and carries no state, there is no change in
28 default behavior.
29 This module provides a few predefined hooks (below) that add useful behavior to
30 $(LREF Expected):
31 
32 $(BOOKTABLE ,
33     $(TR $(TD $(LREF Abort)) $(TD
34         fails every incorrect operation with a call to `assert(0)`.
35 		It is the default third parameter, i.e. `Expected!short` is the same as
36         $(D Expected!(short, string, Abort)).
37     ))
38     $(TR $(TD $(LREF Throw)) $(TD
39         fails every incorrect operation by throwing an exception.
40     ))
41 )
42 
43 The hook's members are looked up statically in a Design by Introspection manner
44 and are all optional. The table below illustrates the members that a hook type
45 may define and their influence over the behavior of the `Checked` type using it.
46 In the table, `hook` is an alias for `Hook` if the type `Hook` does not
47 introduce any state, or an object of type `Hook` otherwise.
48 
49 $(TABLE
50     $(TR $(TH Hook member) $(TH Semantics in Expected!(T, E, Hook))
51     )
52     $(TR $(TD `enableDefaultConstructor`) $(TD If defined, $(LREF Expected) would have
53     enabled or disabled default constructor based on it's `bool` value. Default
54     constructor is disabled by default. `opAssign` for value and error types is
55     generated if default constructor is enabled)
56     )
57     $(TR $(TD `onAccessEmptyValue`) $(TD If value is accessed on unitialized
58     $(LREF Expected) or $(LREF Expected) with error value, $(D hook.onAccessEmptyValue!E(err))
59     is called. If hook doesn't implement the handler, `T.init` is returned.)
60     )
61     $(TR $(TD `onAccessEmptyError`) $(TD If error is accessed on unitialized
62     $(LREF Expected) or $(LREF Expected) with value, $(D hook.onAccessEmptyError())
63     is called. If hook doesn't implement the handler, `E.init` is returned.)
64     )
65 )

81 /++
82 	`Expected!(T, E)` is a type that represents either success or failure.
83 
84 	Type `T` is used for success value.
85 	If `T` is `void`, then $(LREF Expected) can only hold error value and is considered a success when there is no error value.
86 
87 	Type `E` is used for error value.
88 	The default type for the error value is `string`.
89 
90 	Default behavior of $(LREF Expected) can be modified by the `Hook` template parameter.
91 
92 	Params:
93 		T    = represents the expected value
94 		E    = represents the reason explaining why it doesn’t contains avalue of type T, that is the unexpected value.
95 		Hook = defines the $(LREF Expected) type behavior
96 +/

596 /++ Constructs $(LREF Expected) from the result of the provided function.
597 
598 	If the function is `nothrow`, it just returns it's result using $(LREF Expected).
599 
600 	If not, then it uses `try catch` block and constructs $(LREF Expected) with value or error.
601 +/
602 template expected(alias fun, Hook = Abort)
603 {
604 	auto expected(Args...)(auto ref Args args) if (is(typeof(fun(args))))
605 	{
606 		import std.traits : hasFunctionAttributes;
607 
608 		alias T = typeof(fun(args));
609 		static if (is(hasFunctionAttributes!(fun, "nothrow"))) return expected!Exception(fun(args));
610 		else
611 		{
612 			try return Expected!(T, Exception)(fun(args));
613 			catch (Exception ex) return unexpected!T(ex);
614 		}
615 	}
616 }

622 
623 	assert(expected!fn(1) == 1);

665 /++
666 	Returns the error contained within the $(LREF Expected) _and then_ another value if there's no error.
667 	This function can be used for control flow based on $(LREF Expected) values.
668 
669 	Params:
670 		exp = The $(LREF Expected) to call andThen on
671 		value = The value to return if there isn't an error
672 		pred = The predicate to call if the there isn't an error
673 +/
674 auto ref EX andThen(EX)(auto ref EX exp, lazy EX value)
675 	if (is(EX : Expected!(T, E, H), T, E, H))
676 {
677 	return exp.andThen!value;
678 }
679 
680 /// ditto
